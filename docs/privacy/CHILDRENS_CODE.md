# Children's Code Assessment

Status: Draft

| Standard | Latin Quest position | Required evidence/action |
|---|---|---|
| Best interests | Learning-only service with no advertising | Record design decisions and school/pupil consultation |
| DPIA | Draft child-focused DPIA created | Approve before school rollout and review changes |
| Age appropriate application | Secondary-school audience known through school context | Document target ages; do not collect DOB unless necessary |
| Transparency | Full and child-friendly notices provided | Configure real contact and test comprehension with pupils |
| Detrimental use | No sale, advertising or unrelated commercial use | Contractually prohibit and review new features |
| Policies and standards | Privacy/security pack created | Evidence actual operation, training and annual review |
| Default settings | Teacher access scoped to owned classes | Add leaderboard disable/pseudonym option |
| Data minimisation | Limited identity/class/progress data | Review email necessity and answer-retention duration |
| Data sharing | Class and vendor sharing limited by purpose | Complete DPA/subprocessor and export controls |
| Geolocation | Not collected by application | Keep disabled and document any infrastructure IP use |
| Parental controls | Not used | If introduced, make child visibility clear |
| Profiling | Practice ordering, mastery and ranking only | Explain clearly; no significant automated decisions |
| Nudge techniques | Gamification, badges and leaderboard | Test for pressure, shame and excessive engagement risk |
| Connected toys/devices | Not applicable | Reassess if integrations are added |
| Online tools | Rights route via school/operator | Configure privacy email and provide accessible reporting route |

## Design rule

New analytics, AI features, social features, messaging, advertising, geolocation,
parental monitoring or additional profile fields require a DPIA/Children's Code
change review before development and release.
